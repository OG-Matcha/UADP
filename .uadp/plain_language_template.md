## 1. 白話翻譯的目的

### 1.1 目標使用者

- **非技術創業者：** 有想法但不懂技術
- **產品經理：** 需要理解技術決策對產品的影響
- **投資人：** 需要快速理解專案進度

### 1.2 翻譯原則

1. **避免技術術語：** 用日常語言解釋技術概念
2. **說明業務影響：** 解釋變更對使用者的實際影響
3. **提供決策理由：** 說明為什麼選擇這個方案
4. **標示風險等級：** 讓使用者了解風險程度

## 2. 白話翻譯模板

### 2.1 技術變更翻譯模板

#### 模板 A：套件/依賴項變更

```
【技術變更】套件版本調整

【發生了什麼？】
我們把 [套件名稱] 從版本 [舊版本] 更新到 [新版本]。

【為什麼要改？】
因為 [原因，用白話說明，例如：舊版本有安全漏洞 / 新版本有更好的功能 / 與其他套件衝突]。

【對你有什麼影響？】
✅ 好處：[列出好處，例如：更安全 / 功能更多 / 執行更快]
⚠️ 風險：[列出風險，例如：可能有相容性問題 / 需要重新測試]

【風險等級】
🟢 低風險 / 🟡 中風險 / 🔴 高風險

【需要你注意什麼？】
[如果有需要使用者配合的事項，例如：需要重新安裝 / 需要更新設定]
```

**範例：**
```
【技術變更】套件版本調整

【發生了什麼？】
我們把「資料庫連線工具」從版本 2.28.0 降級到 2.27.0。

【為什麼要改？】
因為新版本與另一個工具衝突，導致系統無法正常啟動。

【對你有什麼影響？】
✅ 好處：系統可以正常運作，功能不受影響
⚠️ 風險：無，降級版本是安全的

【風險等級】
🟢 低風險

【需要你注意什麼？】
不需要做任何事，系統會自動處理。
```

```

【為什麼要改？】

【對你有什麼影響？】
✅ 好處：
  - 系統執行更快
  - 可以儲存更多資料
  - 資料更安全（有備份機制）
⚠️ 注意：
  - 舊資料會自動轉換，不需要手動處理
  - 第一次啟動時轉換可能需要幾分鐘

【技術細節】（給技術使用者）
從 JSON 檔案遷移到 SQLite 資料庫，使用 SQLAlchemy ORM。

【風險等級】
🟡 中風險（資料轉換可能有風險，但已測試）

```

```

【為什麼會發生？】

【如何修復？】
我們增加了錯誤處理機制，確保資料一定會正確儲存，即使發生錯誤也會記錄下來。

【對你有什麼影響？】
✅ 現在記帳資料不會再消失了
✅ 如果發生錯誤，系統會顯示錯誤訊息，而不是靜默失敗

【測試結果】
✅ 已通過測試，確認修復成功
```

## 3. 翻譯流程

### 3.1 自動翻譯（推薦）

**在 AUDIT 階段，AI 自動：**
1. 讀取 `amendments.md` 中的所有修正項
2. 讀取技術決策紀錄（`decisions.md`）
3. 根據變更類型選擇對應模板
4. 生成白話翻譯並寫入 `preview_report.md` 或獨立的 `plain_language_report.md`

### 3.2 手動翻譯（備選）

**如果自動翻譯不準確：**
1. AI 提供「翻譯建議」
2. 使用者審查並修正
3. 更新至報告中
